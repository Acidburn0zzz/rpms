This is a proposal to a more standardised naming convention, close to what
Debian and Mandrake already seem to have.

	Interesting links:
		http://www.debian.org/devel/
		http://www.debian.org/doc/debian-policy/ch-sharedlibs.html

		http://www.linux-mandrake.com/en/howtos/mdk-rpm/advanced.html#LIB-POLICY
		http://people.mandrakesoft.com/~warly/files/policy/

		http://moin.conectiva.com.br/PackagingRules

		http://www.freestandards.org/cgi-bin/mailman/listinfo/packaging


Here is a list of packages I already have (or plan to have) and what they've
become.

Packages marked with (x) are not yet changed in my repository.
Packages marked with (%) are packages that are not in my repository.

Normal packages
"""""""""""""""
Packages should be all lowercase (less confusion). If renamed, packages should
provide and obsolete (as a package) their real name too. (so that it becomes
transparant for end-users in apt/yum)

Delimiter for package-parts is the dash '-'. (Not a '_' or a '.')

	hardware-monitor	->	hardware-monitor-applet

Libraries
"""""""""
This is a list of packages that contain shared libraries. Because of that they
should be split (if there are tools too) and renamed. If a package does
contain libraries but they are not used by any other package there's no urge
to move over to this scheme. New packages however should.

Perl packages
"""""""""""""
All perl module packages are prefixed perl-%{name}. (Should they not become
lowercase ? Exception because RH doesn't do it ?), eg.

	Mail-SpamAssassin (perl)->	perl-Mail-SpamAssassin

PHP packages
""""""""""""
All php library packages are prefixed php-%{name}, eg.
